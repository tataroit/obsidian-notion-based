# ИНСТРУКЦИЯ ДЛЯ ИИ: Углубление базы знаний

## Назначение документа

Это руководство для меня (Claude или любой другой ИИ) для работы с **уже существующей** базой знаний. Когда пользователь просит углубить тему, проработать открытые вопросы, или расширить знания — я следую этим правилам для системного углубления.

**Ключевое отличие от создания новой базы:**  
- Документ уже есть, структура есть  
- Задача — не переписать, а **дополнить, углубить, ответить на вопросы**  
- Работа идёт **итерациями** — каждый проход делает документ глубже  
- Think out of the box — искать неочевидные связи, риски, возможности

---

## ▼ ФИЛОСОФИЯ ПРОРАБОТКИ

### Цель углубления

**НЕ** переписать документ заново.

**НЕ** добавить воды или дублировать существующее.

**ДА** найти и закрыть пробелы в понимании.

**ДА** ответить на открытые вопросы конкретно и с источниками.

**ДА** найти то, о чём пользователь **ещё не подумал**, но это критично.

### Принцип работы

**Циклическая проработка:**  
Каждая итерация должна делать документ:  
- **Глубже** — добавлять детали, механику, нюансы  
- **Шире** — охватывать смежные темы, альтернативы, интеграции  
- **Практичнее** — давать конкретные инструкции, цифры, кейсы

**Уровень погружения:**  
- Для специалиста, не для новичка  
- Опускать примитив, фокус на неочевидном  
- Конкретика > абстракция  
- Верифицированные источники > случайные посты

## ▼ ПРОЦЕСС УГЛУБЛЕНИЯ: ПОШАГОВЫЙ АЛГОРИТМ

### Шаг 1: Прочитать весь документ целиком

**Цель:** понять текущее состояние базы знаний.

**Что искать при чтении:**  
1. **Открытые вопросы** — помечены курсивом, обычно в блоках _«Вопросы для дальнейшей проработки»_  
2. **Недостаток конкретики** — есть утверждения без цифр, примеров, механики  
3. **Абстракции** — фразы типа «зависит от ситуации», «может быть», «обычно»  
4. **Пробелы в логике** — упоминается инструмент/подход, но не объяснено как работает  
5. **Устаревшая информация** — даты, регуляции, цены могли измениться  
6. **Отсутствие альтернатив** — описан один способ, но нет сравнения с другими  
7. **Слепые зоны** — темы, которые логично должны быть, но их нет

**Инструмент:**

```Bash
view /path/to/document.md
```

**Вывод этого шага:**  
Список из 5–10 конкретных направлений для углубления.

---

### Шаг 2: Приоритизация направлений

**Не пытаться закрыть всё сразу.** Одна итерация = 2–4 направления максимум.

**Критерии приоритета:**  
1. **Критичность** — влияет на деньги/здоровье/легальность/риски  
2. **Практическая значимость** — пользователь применит это прямо сейчас  
3. **Пробел в базовом понимании** — без этого остальное не работает  
4. **Запрос пользователя** — если пользователь явно указал тему

**Что делать:**  
Выбрать 2–4 вопроса/темы для текущей итерации.

---

### Шаг 3: Исследование каждого направления

Вывести пользователю решение о том, что будет выполнено: какие темы в каких частях будут обновлены

Важно держать пользователя в курсе всех планируемых и совершаемых изменений

### Шаг 4: Исследование каждого направления

**Для каждого выбранного вопроса/темы:**

### 3.1. Определить тип вопроса

|Тип вопроса|Что нужно|Инструмент|
|---|---|---|
|Фактическая информация (цены, даты, законы)|Актуальные данные|web_search|
|Механика процесса (как работает X)|Детальное объяснение|web_search + синтез|
|Сравнение вариантов|Таблица с параметрами|Структурированный анализ|
|Интеграционный риск|Кейсы, практика, подводные камни|web_search + логический вывод|
|Юридический аспект|Законы, статьи, судебная практика|web_search (gov сайты, юр. базы)|

### 4.2. Провести исследование

**Правила исследования:**  
- Использовать **верифицированные источники**: gov, официальные документы компаний, профильные СМИ (не блоги)  
- Для технических тем: официальная документация, GitHub, Stack Overflow (проверенные ответы)  
- Для финансов: отчёты компаний, SEC filings, данные бирж  
- Для законов: сайты ФНС, Consultant.ru, судебная практика

**Инструменты:**

```Bash
web_search "конкретный запрос"
web_fetch "https://проверенный-источник.com/article"
```

**Критерий достаточности:**  
- Найдено 2–3 независимых источника, подтверждающих информацию  
- Есть конкретные цифры/даты/названия  
- Понятна механика (не просто «что», а «как» и «почему»)

### 4.3. Синтез и проверка

**Перед добавлением в документ:**  
- Проверить актуальность (дата источника, релевантность для 2025+)  
- Убрать противоречия (если источники расходятся — указать это явно)  
- Упростить до конкретики (без «может быть», только факты)

### Шаг 5: Интеграция в документ

**Принципы интеграции:**

### 5.1. Не дублировать

Перед добавлением — проверить, нет ли этой информации уже в документе.

Если есть похожее — **дополнить**, а не написать заново.

**Плохо:**

```Markdown
## Раздел про налоги
[старый текст]

## Раздел про налоги (новая информация)
[почти то же самое, но чуть иначе]
```

**Хорошо:**

```Markdown
## Раздел про налоги
[старый текст]

**Дополнение (2025):**
[новая конкретная информация, которая расширяет понимание]
```

### 5.2. Сохранить структуру

Добавлять информацию в **правильный раздел**.

Если раздела нет — создать подраздел, но не нарушать общую иерархию.

**Формат добавления:**

```Markdown
### [Существующий подраздел]

[Старый контент]

**[Название нового аспекта]:**
[Новая информация с конкретикой]

**Конкретные данные:**
-[Цифра/факт из источника 1]
-[Цифра/факт из источника 2]

*Источник: [ссылка на верифицированный источник]*
```

### 5.3. Закрыть открытый вопрос

Если отвечаешь на вопрос из блока _«Вопросы для дальнейшей проработки»_:

**Шаг 1:** Найти блок с вопросом

**Шаг 2:** Написать ответ прямо под вопросом или в соответствующем разделе

**Шаг 3:** Удалить вопрос из блока или пометить как закрытый

**Пример:**

**Было:**

```Markdown
*Вопросы для дальнейшей проработки:*
-Какой банк в Казахстане даёт UnionPay без резидентства? Процедура, стоимость?
-Как работает mNAV для MSTR?
```

**Стало:**

```Markdown
**UnionPay в Казахстане без резидентства:**

По состоянию на февраль 2025:
-**Halyk Bank** — выдаёт UnionPay нерезидентам при личном визите
-**Процедура:** паспорт + адрес регистрации в любой стране + 50,000 тенге депозит
-**Стоимость:** 15,000 тенге за выпуск, 3,000 тенге/год обслуживание
-**Срок:** 2–3 рабочих дня

*Источник: официальный сайт Halyk Bank, февраль 2025*

*Вопросы для дальнейшей проработки:*
-~~Какой банк в Казахстане даёт UnionPay без резидентства? Процедура, стоимость?~~ ✅ Закрыт
-Как работает mNAV для MSTR?
```

### 5.4. Добавить новые вопросы

В процессе исследования часто возникают **новые** вопросы.

**Добавлять их в блок открытых вопросов:**

```Markdown
*Вопросы для дальнейшей проработки:*
-[Новый вопрос, который возник при исследовании старого]
```

### Шаг 6: Расширение смежных областей (think out of the box)

**После ответа на прямой вопрос — подумать:**

### 6.1. Интеграционные риски

«Если пользователь применит это, что ещё может пойти не так?»

**Примеры:**  
- Вопрос был про покупку крипты через Currenty  
- Ответ: как покупать, комиссии, лимиты  
- **Дополнительно подумать:** а что если купленная там крипта будет иметь высокий taint? Заблокируют ли на Coinbase? Как проверить?

**Добавить в документ:**

```Markdown
**Интеграционный риск:**
Крипта с белорусских платформ может иметь повышенный taint score на западных биржах. Проверять через AMLBot перед отправкой на Coinbase/Kraken.
```

### 6.2. Альтернативы

«Есть ли другой способ решить ту же задачу?»

**Если в документе описан один метод — найти и добавить альтернативные.**

**Пример:**  
- Есть: купить BTC через Binance P2P  
- Добавить: купить BTC через Currenty (другая комиссия, другие риски)  
- Сравнительная таблица

### 6.3. Граничные случаи

«Что происходит в экстремальных ситуациях?»

**Примеры:**  
- Вопрос про налоги на крипту  
- Ответ: как платить НДФЛ  
- **Дополнительно:** что если доход >10 млн руб? Что если крипта получена в наследство? Что если потерял ключи — можно ли списать как убыток?

**Добавить в документ:**

```Markdown
**Граничные случаи:**
-Доход >10 млн руб/год: может потребоваться аудит, усиленная проверка ФНС
-Наследство: крипта = имущество, наследуется, налог на наследство для близких родственников — 0%
-Потеря ключей: нельзя списать как убыток для налоговых целей (нет подтверждающего документа)
```

### 6.4. Слепые зоны

«О чём пользователь **точно не подумал**, но это важно?»

**Примеры:**  
- Тема: покупка MSTR как замена BTC  
- Добавить: а что при разводе? MSTR = имущество, делится 50/50. Брачный договор?

### Шаг 7: Проверка качества добавленного контента

**Перед сохранением — проверить каждый новый блок:**

- [ ] **Конкретика:** есть цифры, даты, названия (не абстракции)

- [ ] **Источник:** указан верифицированный источник (или логический вывод помечен как таковой)

- [ ] **Актуальность:** информация свежая (2024–2025)

- [ ] **Практичность:** пользователь может применить это прямо сейчас

- [ ] **Нет воды:** каждое предложение несёт смысл

- [ ] **Нет дублирования:** этого нет в документе в другом месте

- [ ] **Связность:** новый блок логично встроен в существующую структуру

## ▼ ТИПОВЫЕ НАПРАВЛЕНИЯ УГЛУБЛЕНИЯ

### 1. Ответы на открытые вопросы

**Приоритет:** высокий (это явно обозначенные пробелы)

**Подход:**  
- Найти все блоки _«Вопросы для дальнейшей проработки»_  
- Выбрать 2–4 вопроса  
- Исследовать через web_search  
- Ответить конкретно с источниками  
- Пометить как закрытые

---

### 2. Конкретизация абстракций

**Признак:** фразы «зависит от», «обычно», «может быть», «в большинстве случаев»

**Подход:**  
- Найти такие фразы  
- Заменить конкретикой:  
- «Зависит от суммы» → «До $10k — комиссия 3%, от $10k до $100k — 2%, свыше $100k — 1.5%»  
- «Обычно занимает время» → «Обработка 2–5 рабочих дней»

---

### 3. Добавление сравнительных таблиц

**Когда нужно:**  
- В документе описаны 3+ варианта, но нет сравнения  
- Есть альтернативы, но не ясно когда что использовать

**Подход:**  
- Создать таблицу с параметрами: стоимость, время, риски, для кого  
- Добавить вывод после таблицы

---

### 4. Обновление устаревшей информации

**Признаки устаревания:**  
- Даты старше 1 года (для быстро меняющихся тем)  
- Упоминание закрытых сервисов  
- Цены/курсы/лимиты, которые могли измениться

**Подход:**  
- Найти через web_search актуальные данные  
- Обновить с пометкой «По состоянию на [дата]»  
- Если данные изменились значительно — добавить примечание

**Пример:**

```Markdown
**Комиссия Currenty (обновлено февраль 2025):**
-Карта: 4–5% (было 3.5–5% в 2024)
-Банковский перевод: 2% (без изменений)

*Рост комиссии связан с увеличением стоимости эквайринга.*
```

---

### 5. Расширение практических кейсов

**Когда нужно:**  
- Есть теория, но мало практики  
- Нет примеров «как сделать пошагово»

**Подход:**  
- Добавить раздел «Пошаговая инструкция для [профиль пользователя]»  
- Конкретные шаги с ожидаемым результатом каждого

**Пример:**

```Markdown
### Пошаговая инструкция: первая покупка BTC через Currenty

**Профиль:** новичок, сумма 100 тыс руб, карта Т-Банка

1.Регистрация на Currenty.io
   -Указать email, придумать пароль
   -Подтвердить email (письмо приходит в течение 5 мин)

2.KYC (верификация)
   -Загрузить скан паспорта РФ (1-й разворот)
   -Сделать селфи с паспортом
   -Ожидание проверки: 1–24 часа

3.Пополнение счёта
   -Выбрать «Пополнить» → «Банковская карта»
   -Ввести сумму: 100,000 руб
   -Комиссия: ~4,500 руб (4.5%)
   -К зачислению: 95,500 руб

4.Покупка BTC
   -Курс на момент покупки: 1 BTC = 6,200,000 руб
   -Покупаешь: 0.0154 BTC

5.Вывод на свой кошелёк
   -Указать адрес своего BTC-кошелька
   -Комиссия сети: ~$3–5 (зависит от загрузки)
   -Время: 10–60 минут

**Итого:** получил 0.0154 BTC на свой кошелёк за 100,000 руб (эффективная комиссия ~4.5%)
```

### 6. Добавление интеграционных рисков

**Что искать:**  
- Взаимодействие между разными частями системы  
- «Если сделать A, то что произойдёт с B?»

**Примеры тем:**  
- Купил крипту на российской бирже → хочу перевести на западную — что может пойти не так?  
- Открыл счёт в Казахстане → как это влияет на налоги в РФ?  
- Использую VPN для Binance → могут ли заблокировать аккаунт?

**Добавить в документ:**

```Markdown
### Интеграционные риски

**Российская биржа → западная биржа:**
-Coinbase/Kraken могут запросить source of funds
-EXMO, Garantex считаются источниками повышенного риска
-Возможна блокировка аккаунта до предоставления объяснений
-**Решение:** перевести сначала на личный кошелёк, подождать 1–2 недели, потом на западную биржу
```

---

### 7. Добавление слепых зон

**Что это:**  
Темы, о которых пользователь не спросил, но они критичны.

**Как искать:**  
- Подумать про edge cases (граничные ситуации)  
- Что происходит при масштабировании?  
- Какие юридические последствия?  
- Что если пользователь умер/развёлся/уехал из страны?

**Примеры для крипты:**  
- Наследование: как передать seed-фразу наследникам?  
- Развод: крипта = имущество, делится  
- Потеря seed: можно ли списать как убыток для налогов? (нет)  
- Утечка KYC: что делать если паспортные данные утекли с биржи?

**Добавить раздел:**

```Markdown
### Чего ещё не учёл (слепые зоны)

#### 1. Наследование криптоактивов
[детальный разбор]

#### 2. Развод и раздел имущества
[детальный разбор]
```

## ▼ РАБОТА С ИСТОЧНИКАМИ

### Приоритет источников

**Tier 1 (максимальный приоритет):**  
- Официальные сайты государственных органов (.gov, nalog.ru)  
- Законы и кодексы (consultant.ru, официальные тексты)  
- Официальная документация компаний (investor relations, SEC filings)  
- Peer-reviewed исследования (SciHub, Google Scholar)

**Tier 2 (хорошо):**  
- Профильные СМИ с репутацией (CoinDesk для крипты, Investopedia для финансов)  
- Официальные блоги компаний  
- Проверенные YouTube-каналы экспертов (с подтверждёнными credentials)

**Tier 3 (с осторожностью):**  
- Форумы (Reddit, BitcoinTalk) — только если информация подтверждена другими источниками  
- Личные блоги экспертов — проверить credentials автора  
- Telegram-каналы — только факты, не мнения

**Tier 4 (не использовать):**  
- Случайные посты в соц.сетях  
- Сомнительные «гуру» типа Роберта Кийосаки  
- Источники с явным конфликтом интересов (реклама продукта под видом обзора)

### Как оформлять источники

**В тексте:**

```Markdown
**[Факт с конкретикой]**

*Источник: [название источника], [дата], [ссылка если онлайн]*
```

**Пример:**

```Markdown
**Комиссия Currenty:** 4–5% при оплате картой РФ (февраль 2025)

*Источник: официальный сайт Currenty.io, тарифы, февраль 2025*
```

**Если источник — логический вывод:**

```Markdown
**[Вывод]**

*Логический вывод на основе [X] и [Y]*
```

## ▼ ТИПОВЫЕ ОШИБКИ ПРИ УГЛУБЛЕНИИ

### Ошибка 1: Переписать вместо дополнить

**Симптом:**  
Удалил старый раздел, написал новый на ту же тему.

**Решение:**  
Читать что уже есть. Дополнять, а не заменять (если только старое не ошибочно).

---

### Ошибка 2: Добавить воду

**Симптом:**  
Новый раздел на 10 абзацев, а конкретики в нём — 2 цифры.

**Решение:**  
Каждое предложение должно нести конкретную информацию. Убрать всё лишнее.

---

### Ошибка 3: Не закрыть открытый вопрос

**Симптом:**  
Исследовал тему, написал новый раздел, но вопрос остался в списке открытых.

**Решение:**  
После ответа на вопрос — пометить как закрытый или удалить из списка.

---

### Ошибка 4: Дублирование информации

**Симптом:**  
Один и тот же факт описан в двух разных разделах.

**Решение:**  
Перед добавлением — поиск по документу (Ctrl+F). Если есть — дополнить существующее место.

---

### Ошибка 5: Нет источника для фактов

**Симптом:**  
Написал конкретные цифры, но не указал откуда.

**Решение:**  
Для каждого факта — источник. Если нет источника — пометить как «требует проверки».
